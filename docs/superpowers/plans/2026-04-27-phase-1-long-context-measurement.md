# Phase 1: Long-Context Measurement Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace Java's `chars/4` heuristic with an authoritative Claude token count for the NCC 2022 Vol 2 corpus, and write a Python-side findings doc that closes Phase 1.

**Architecture:** A standalone `scripts/measure_tokens.py` script loads the corpus PDF, extracts text, and queries Anthropic's free `messages.count_tokens` endpoint. No FastAPI changes, no new modules under `src/knowledge_bot/`, no automated tests (per spec — measurement spike, not running infrastructure).

**Tech Stack:** Python 3.12, `uv`, `pypdf` (PDF extraction), `anthropic` SDK (token counting), `python-dotenv` (env loading — already a Phase 0 dep).

**Spec:** `docs/superpowers/specs/2026-04-27-phase-1-long-context-measurement-design.md`

```python
"""Phase 1 measurement: count tokens in the NCC 2022 Vol 2 corpus.

Confirms via Anthropic's authoritative tokenizer that the corpus exceeds
Claude's 200K context window. Output goes to stdout; results land in
docs/observations/phase-1-findings.md.

Run: uv run python scripts/measure_tokens.py
"""
from __future__ import annotations

import os
import sys
import time
from pathlib import Path

import anthropic
from dotenv import load_dotenv
from pypdf import PdfReader

CORPUS_PATH = Path("corpus/ncc-2022-vol2.pdf")
MODEL = "claude-haiku-4-5"
WINDOW_TOKENS = 200_000
HEADROOM_TOKENS = 10_000  # system prompt + question + answer
CEILING_TOKENS = WINDOW_TOKENS - HEADROOM_TOKENS


def load_pdf_text(path: Path) -> tuple[str, int, float]:
    """Extract concatenated text from every page of the PDF.

    Returns (text, page_count, load_seconds).
    """
    if not path.exists():
        sys.exit(
            f"Corpus not found at {path}. "
            f"See README setup section: copy NCC 2022 Vol 2 PDF into corpus/."
        )
    start = time.perf_counter()
    reader = PdfReader(str(path))
    pages = [page.extract_text() or "" for page in reader.pages]
    text = "\n".join(pages)
    elapsed = time.perf_counter() - start
    return text, len(reader.pages), elapsed


def count_claude_tokens(text: str) -> int:
    """Authoritative token count via Anthropic's count_tokens endpoint."""
    client = anthropic.Anthropic()
    response = client.messages.count_tokens(
        model=MODEL,
        messages=[{"role": "user", "content": text}],
    )
    return response.input_tokens


def print_results(
    *,
    page_count: int,
    char_count: int,
    chars_div_4: int,
    anthropic_tokens: int,
    load_seconds: float,
) -> None:
    rows = [
        ("Document", "NCC 2022 Volume 2 (Housing Provisions)"),
        ("Pages", f"{page_count}"),
        ("Characters", f"{char_count:,}"),
        ("chars / 4 estimate", f"{chars_div_4:,}"),
        (f"Anthropic count_tokens ({MODEL})", f"{anthropic_tokens:,}"),
        ("Load time (pypdf)", f"{load_seconds:.2f}s"),
        ("% of 200K window", f"{anthropic_tokens / WINDOW_TOKENS:.1%}"),
        ("% of 190K ceiling", f"{anthropic_tokens / CEILING_TOKENS:.1%}"),
    ]
    width = max(len(k) for k, _ in rows)
    print()
    for k, v in rows:
        print(f"  {k.ljust(width)}  {v}")
    print()


def main() -> None:
    load_dotenv()
    if not os.getenv("ANTHROPIC_API_KEY"):
        sys.exit("ANTHROPIC_API_KEY missing. Set it in .env or the environment.")

    text, page_count, load_seconds = load_pdf_text(CORPUS_PATH)
    char_count = len(text)
    chars_div_4 = char_count // 4
    anthropic_tokens = count_claude_tokens(text)

    print_results(
        page_count=page_count,
        char_count=char_count,
        chars_div_4=chars_div_4,
        anthropic_tokens=anthropic_tokens,
        load_seconds=load_seconds,
    )


if __name__ == "__main__":
    main()
```

# Phase 1 Findings: Long-Context Measurement (Python)

## Corpus measurement

| Metric | Value |
|---|---|
| Document | NCC 2022 Volume 2 (Housing Provisions) |
| Pages | <PAGES> |
| Characters | <CHARS> |
| chars / 4 estimate | <CHARS_DIV_4> |
| Anthropic `count_tokens` (claude-haiku-4-5) | <ANTHROPIC_TOKENS> |
| Load time (pypdf) | <LOAD_SECONDS>s |

## Finding: long-context approach is impossible for this corpus

Claude's context window is 200K tokens. Leaving 10K headroom for the system prompt, question, and answer leaves a corpus ceiling of 190K tokens.

Anthropic's `count_tokens` endpoint reports the full NCC Vol 2 at **<ANTHROPIC_TOKENS> tokens — <RATIO>× the 200K window**. There is no way to fit this corpus into a single Claude prompt. The chars/4 heuristic estimated <CHARS_DIV_4> tokens; the authoritative count was <DELTA_PERCENT>% <higher|lower>, which does not move the conclusion — the gap is structural, not an estimation artefact.

## Why not use a subset?

The Java repo's Phase 1 verified that Part H4 — Health and amenity (pages 117–125) extracts cleanly at ~18K tokens, well within budget. But running an `/ask` endpoint against 18K tokens does not produce a meaningful long-context baseline:

- 18K tokens is trivially small — any LLM handles it with near-perfect recall.
- The baseline was supposed to measure the *pain* of long context (high token cost, latency, degraded recall). With only 18K tokens, there is no pain to measure.
- Phase 2's RAG cost / latency improvement over Phase 1 would be marginal, undermining the motivation story.

The real Phase 1 finding is not a set of latency numbers — it is that the approach **fails by construction** for a real-world corpus of this size.

## Cross-stack note

The [Java repo's Phase 1 findings](https://github.com/Harryqin99/build-knowledge-bot/blob/main/docs/observations/phase-1-findings.md) reached the same conclusion via the chars/4 heuristic, which estimated ~607K tokens. This Python phase replaced the heuristic with Anthropic's `count_tokens` endpoint and recorded an exact count of <ANTHROPIC_TOKENS> tokens — a <DELTA_PERCENT>% <over|under>-estimate.

The 3× gap between corpus size and context window holds regardless of which tokenizer measures it. The two repos converge on the same forward-pointer to RAG: chunking, embeddings, and retrieval are not optional optimisations for a corpus this size — they are the only path forward.

## What this means for Phase 2

1. **Chunking + embeddings** break the corpus into pieces small enough to fit any context window.
2. **Retrieval** selects only the relevant chunks per question — token usage drops from "entire corpus" to "a few relevant passages".
3. **The `/ask` endpoint** (FastAPI route, Anthropic call, response shape) gets built in Phase 2, wired to RAG from the start rather than long context.
